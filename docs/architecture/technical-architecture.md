## 1. 架构目标

- 微信小程序优先上线，同时保持支付宝小程序、抖音小程序和 H5 的可移植性。
- 将平台能力与业务逻辑隔离，避免微信专有 API 扩散到业务组件。
- 对宝宝照片、家庭关系和成长数据实施服务端强制权限控制。
- 支持后续增加记录类型，而不破坏现有时间轴和导出能力。
- MVP 保持单体服务，避免过早引入微服务复杂度。

## 2. 总体架构

```text
微信/支付宝/抖音小程序、H5
            │
       Taro 客户端
            │ HTTPS JSON API
       API 网关/反向代理
            │
      NestJS 单体应用
       ├── 认证与账号
       ├── 宝宝与家庭
       ├── 成长记录
       ├── 文件上传
       └── 数据导出
        │          │
  PostgreSQL   对象存储
        │          │
     备份       CDN/临时签名地址
```

## 3. 技术选型

### 3.1 客户端

| 类别 | 建议选型 | 说明 |
| --- | --- | --- |
| 跨端框架 | Taro 4 | React 生态成熟，支持目标小程序和 H5 |
| UI | React + TypeScript | 统一类型和组件模型 |
| 状态管理 | Zustand | 管理会话、当前宝宝和少量全局状态 |
| 服务端状态 | TanStack Query 或轻量请求封装 | 缓存、失效、重试和分页；初始化工程时二选一确认 |
| 样式 | Sass Modules 或项目统一 Sass 约定 | 减少全局样式污染 |
| 图表 | ECharts Taro 兼容封装 | 需在微信真机验证包体和渲染性能 |
| 表单校验 | Zod + 自定义表单层 | 复用类型和校验规则 |
| 测试 | Vitest + React Testing Library | 单元和组件测试 |

### 3.2 服务端

| 类别 | 建议选型 | 说明 |
| --- | --- | --- |
| 运行时 | Node.js LTS | 与客户端共享 TypeScript 能力 |
| 框架 | NestJS | 模块、守卫、验证和 OpenAPI 支持完善 |
| ORM | Prisma | 数据迁移和类型生成清晰 |
| 数据库 | PostgreSQL | 事务、约束、索引和 JSON 扩展能力完整 |
| 缓存/队列 | MVP 初期不强制 Redis | 导出任务量增加后再引入队列 |
| 对象存储 | S3 兼容服务 | 使用预签名上传和限时读取地址 |
| 接口文档 | OpenAPI 3 | 从 NestJS 注解生成并纳入版本控制 |
| 测试 | Vitest/Jest + Supertest | 服务和接口测试 |

## 5. 客户端分层

### 5.1 页面层

- 负责路由参数、页面布局和页面级状态组合。
- 不直接调用 Taro 平台 API，统一通过 `platform` 层。
- 不在页面中拼接接口地址或处理认证头。

### 5.2 功能层

以业务域组织组件、hooks、校验和局部状态：

- `features/auth`
- `features/babies`
- `features/timeline`
- `features/growth`
- `features/family`
- `features/exports`

### 5.3 平台适配层

统一定义能力接口，例如：

- `login()`：获取平台临时登录凭证。
- `share()`：生成或触发平台分享。
- `chooseImages()`：选择或拍摄图片。
- `getStorage()/setStorage()`：本地存储。
- `getSystemInfo()`：安全区域和设备信息。

不同构建目标提供各自实现。业务代码不得直接调用 `wx.*`。

### 5.4 API 层

- 统一 base URL、认证头、超时和错误映射。
- 收到 401 时清理失效会话并执行一次可控的重新登录，不无限重试。
- 写接口支持 `Idempotency-Key`。
- 服务端时间统一使用 ISO 8601 UTC，界面按用户时区展示。

## 6. 服务端模块

| 模块 | 职责 |
| --- | --- |
| Auth | 平台凭证交换、业务会话、账号绑定 |
| Users | 用户资料和账号状态 |
| Babies | 宝宝档案、当前用户可访问宝宝列表 |
| Families | 成员、角色、邀请、移除和权限检查 |
| Records | 图文、里程碑和测量记录的统一生命周期 |
| Media | 上传凭证、文件元数据、访问签名、孤儿文件清理 |
| Exports | 创建导出任务、生成文件、状态和下载地址 |
| Audit | 敏感操作审计；MVP 可落在同一数据库 |

## 7. 认证与授权

### 7.1 微信登录

```text
客户端调用 Taro.login 获取 code
→ 客户端提交平台类型和 code
→ 服务端向微信换取 openid/session_key
→ 查找或创建 platform_identity 和 user
→ 服务端签发短期访问令牌与可轮换刷新令牌
→ 客户端安全存储业务令牌
```

- 不将微信 `session_key` 返回客户端。
- 不使用 `openid` 作为对外业务主键。
- 其他平台通过相同身份接口接入 `platform_identity`。
- H5 登录方式待确认，可采用微信 OAuth 或手机/邮箱绑定。

### 7.2 授权模型

所有宝宝域资源通过 `baby_members` 判断权限：

1. 验证用户会话。
2. 从路由或资源反查 `baby_id`。
3. 查询有效成员关系。
4. 根据角色和资源创建者判断操作权限。
5. 对敏感操作写入审计日志。

任何客户端传入的角色信息都不可信。

## 8. 成长记录建模策略

使用统一 `records` 主表承载所有记录共有信息，测量专有字段放入 `measurement_records`，媒体通过关联表连接。

优点：

- 时间轴可统一分页和排序。
- 权限、软删除、创建者和审计逻辑只实现一次。
- 后续增加睡眠、喂养等类型时可新增详情表。

不建议将所有类型字段放入单个无约束 JSON 字段；少量展示扩展可以使用 `metadata`，核心数值仍使用强类型列和约束。

## 9. 图片上传与访问

推荐直传对象存储：

```text
客户端申请上传凭证
→ 服务端校验家庭权限并创建 media 占位记录
→ 客户端使用预签名地址上传对象存储
→ 客户端通知服务端完成
→ 服务端校验文件存在、大小和类型
→ 创建成长记录并关联 media
```

安全要求：

- 对象默认私有，不暴露永久公共 URL。
- 对图片类型、大小、数量进行服务端校验。
- 对象键由服务端生成，不接受用户提供完整路径。
- 未关联的上传在固定时间后清理。
- 读取时返回短期签名 URL，导出任务使用独立的限时地址。
- 内容安全审核方案需结合首发平台要求确认。

## 10. API 设计约定

- 基础路径：`/api/v1`。
- 请求和响应使用 JSON；文件使用预签名上传。
- 资源标识使用不可枚举 UUID。
- 列表优先采用游标分页：`cursor`、`limit`、`nextCursor`。
- 时间字段为 ISO 8601 字符串。
- 成功响应直接返回资源或稳定的数据包装结构，项目初始化时统一。
- 错误至少包含 `code`、`message`、`requestId`，字段错误包含 `details`。
- 写操作使用事务保证主记录、详情和媒体关联一致。
- OpenAPI 文件是接口事实来源，需纳入 CI 校验。

建议错误码：

| 错误码 | 含义 |
| --- | --- |
| `AUTH_REQUIRED` | 未登录或会话失效 |
| `FORBIDDEN` | 没有对应宝宝或操作权限 |
| `RESOURCE_NOT_FOUND` | 资源不存在或对用户不可见 |
| `VALIDATION_FAILED` | 输入校验失败 |
| `INVITE_INVALID` | 邀请无效、过期、撤销或已使用 |
| `UPLOAD_INCOMPLETE` | 文件尚未完成或校验失败 |
| `CONFLICT` | 重复成员、版本冲突等 |
| `RATE_LIMITED` | 操作过于频繁 |

## 11. 幂等、并发与软删除

- 创建记录、接受邀请和创建导出任务支持幂等键。
- 成员邀请接受过程在数据库事务内完成，并对唯一约束冲突做可读映射。
- 可编辑资源建议使用 `updated_at` 或版本号处理覆盖冲突。
- 宝宝、记录、成员关系采用软删除或状态字段；查询默认排除失效数据。
- 对象存储文件在业务软删除后延迟物理清理，避免误删后无法恢复。

## 12. 安全与隐私

- 全链路 HTTPS。
- 数据库与对象存储启用静态加密和定期备份。
- 日志不记录令牌、平台凭证、宝宝姓名、正文和签名 URL。
- 接口对登录、邀请、上传和导出操作限流。
- 登录、模拟登录、邀请预览/创建/接受及媒体上传授权由应用内按可信代理解析后的
  来源 IP 限流；该实现不读取客户端角色或未验证的 bearer token。应用内计数仅作为单实例
  保护，staging/production 多实例部署必须同时在可信网关配置共享限流，
  且不得信任未经 `TRUST_PROXY` 明确声明的转发来源。
- 生产环境密钥使用云密钥管理或平台 Secret，不写入仓库。
- 数据导出地址短期有效，下载行为记录审计日志。
- 用户注销、宝宝删除和数据保留期限在隐私政策中明确。

### 12.1 HTTP 与运行配置基线

- JSON 请求体上限由 `JSON_BODY_LIMIT_BYTES` 控制，默认 256 KiB；媒体仍经
  私有对象存储预签名直传，不通过 API JSON 传输。
- API 返回 CSP、nosniff、禁止 iframe、no-referrer、Permissions Policy
  和 no-store；staging/production 额外返回 HSTS。
- 运行日志只记录 Nest/Express 匹配后的规范化路由模板，禁止记录实际 UUID、
  query、请求体或签名 URL；5xx 只记录安全错误类型和稳定 API 错误码。
- Swagger 默认关闭；本地可显式开启，staging/production 如确需开启必须由
  `x-internal-monitoring-token` 保护。该内部 token 不得复用 JWT 或存储凭据。
- staging/production 启动时强制校验微信 AppID/AppSecret、微信和 S3 HTTPS
  端点；production 的 PostgreSQL 连接必须显式配置 TLS `sslmode`。

## 13. 可观测性

- 每个请求生成 `requestId`，客户端错误反馈可携带该 ID。
- 结构化日志包含环境、模块、接口、耗时和状态码。
- 关键指标：接口错误率、P95 延迟、登录失败率、上传失败率、导出失败率。
- 异常告警不得携带敏感业务正文。
- 审计日志与运行日志分开保存。

## 14. 环境与发布

建议环境：

- `local`：本机开发。
- `test`：自动化和联调。
- `staging`：微信体验版与发布前验收。
- `production`：正式环境。

每个环境使用独立数据库、对象存储前缀或桶及平台配置。数据库变更通过 Prisma migration 执行，不允许在生产环境手工改表。

### 14.1 本地优先开发

本地环境不依赖任何具体云服务商：

- PostgreSQL 通过 Docker Compose 启动。
- 对象存储使用 MinIO，并通过 S3 兼容适配器访问。
- API 运行在本机 Node.js 环境。
- 客户端可先使用 Taro H5 验证通用流程，再使用微信开发者工具验证小程序构建。
- 登录使用仅在 `local` 和 `test` 环境开放的模拟身份接口。
- 本地导出任务使用同一 API 进程内的后台 worker 或独立本地 worker。

模拟登录路由必须通过环境变量显式启用，并在 `staging` 和 `production` 启动时强制拒绝该配置。

### 14.2 云资源确定时间

M0 只确定 PostgreSQL、S3 兼容对象存储、HTTPS 和私有文件访问等能力边界。具体云厂商、部署区域、CDN 和托管产品应在预发布环境建设前确定，不阻塞 M1 至 M4 的本地开发。

## 15. 测试策略

- 单元测试：权限规则、年龄/月龄计算、校验、时间转换。
- 数据库集成测试：成员唯一性、邀请并发、软删除和事务回滚。
- API 测试：登录、档案、记录、成员、上传和导出流程。
- 客户端组件测试：表单校验、权限状态和失败重试。
- 端到端测试：首次登录、添加记录、查看趋势、邀请成员和切换宝宝。
- 微信开发者工具与至少一台真机完成核心流程验收。

## 16. 待确认事项

- 云服务商、部署区域、对象存储和 CDN 产品；最晚在预发布环境建设前确定，不阻塞本地开发。
- H5 身份认证方式。
- 图片内容安全审核产品与正式拦截策略。

已实现决策：客户端使用 `useSyncExternalStore` 轻量 store 和按宝宝隔离的请求上下文；
成长图表使用小型跨端组件而不引入 ECharts；导出使用数据库任务表和独立常驻 worker；
图片只保留一份实际上传对象，不另存原图/压缩图/缩略图组合。
